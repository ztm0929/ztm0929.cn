# 开发与优化指南

本文档为本项目的网站开发规范与优化方向的纲要，供贡献者和维护者参考。内容以可执行、易验证的实践为主，保持精简并可随时演进。

## 目标

- 明确代码风格与提交规范，保证可维护性。
- 建立可复现的本地开发流程与 CI/CD 流水线。
- 优化构建与运行性能，提升页面加载体验与 SEO。
- 建立监控与错误跟踪，确保线上可观测性。

## 代码风格与约定

- 使用 TypeScript 严格模式（若尚未启用，建议开启）。
- 代码格式：使用 `prettier` 统一格式化；在提交前运行格式化检查。
- Lint：使用 `eslint` 并集成到 CI，确保无严重类型/风格违规。
- 文件/目录命名遵循已有约定（例如 `content/*`, `src/*`），避免破坏路由结构。

## 内容管理

- 文档与博客使用 MDX/MD 存放在 `content/` 下，frontmatter 应遵循现有 schema（`frontmatterSchema`）。
- 新增内容应同时更新相关 meta（例如 `content/docs/meta.json`）以维护侧边栏与聚合。
- 草稿使用 `draft: true`，生产构建需排除草稿。

## 本地开发流程

- 推荐运行：

```powershell
pnpm install
pnpm dev
```

- 遇到构建或环境异常，先检查 `package.json` 脚本与 `.env` 配置（若有）。

## 构建与部署

- 使用 `pnpm build` 做生产构建并验证：

```powershell
pnpm build
pnpm start
```

- 在 CI 中至少包含：安装依赖、类型检查、lint、构建、简单静态校验（如检查未发布的 draft）。

## 性能优化方向

- 图片与媒体：使用现代格式（AVIF/WebP），并在可能时启用按需加载与合理的占位。
- 静态资源缓存：在部署时设置长期缓存与变更后缀（cache-busting）。
- 页面加载：优先 SSR/SSG，对高频访问页面采用静态生成以降低首次加载成本。
- 代码分割：避免在根页面引入大型依赖，按路由或组件延迟加载。

## 可访问性与 SEO

- 保持语义化 HTML，确保主要页面通过 Lighthouse 无重大可访问性违规。
- meta 标签、Open Graph、结构化数据应在模板或页面 frontmatter 中定义。

## 测试与验证

- 类型安全：在 CI 中强制 `pnpm -s tsc --noEmit`（或等效命令）。
- 端到端/集成测试：对重要页面与 API 路由添加基础测试（可选，推荐使用 Playwright 或类似工具）。

## 监控与错误跟踪

- 建议接入错误收集（例如 Sentry）与简单的性能监控（如页面视图/核心指标）。
- 部署后保留最近 30 天的错误日志以便回溯。

## 合并与 PR 流程

- 所有非小修应通过 Pull Request；PR 应包含变更说明与预期影响。
- CI 必须通过后方可合并；对重大改动建议至少一名审阅者批准。

## 未来优化方向（可迭代）

- 自动化内容提要（RSS / JSON）与更完善的搜索索引。
- 引入图片/资源构建管线（自动生成多分辨率、压缩流程）。
- 性能基线与回归检测：在 CI 中加入 Lighthouse 基线测试。

## 参考与检查清单

- 在开始新功能或优化前，复查：
  - 是否需要新增 `content` 条目或更改路由？
  - 是否需更新 `source.config.ts` 或 `src/lib/source.ts` 的 loader？
  - 是否会影响 SEO 或站点结构？
